## PART 1: BUGS WE FIXED (V1.3.5 Session)

These are the 7 commits worth of fixes applied during the V1.3.5 development session. Each one was a real bug that was silently corrupting backtest results or live behavior.

### 1.1 Commission Was Per-Leg, Not Per-Contract (CRITICAL)

**Commit:** `2c77080f`
**What was wrong:** Commission was calculated as `$1.29 × 2 × number_of_legs` instead of `$1.29 × 2 × number_of_contracts`. A 5-contract trade with 3 legs was charged $7.74 instead of $12.90.
**Impact:** Every single trade in backtest history was undercharged by ~40% on commission. Net PnL was inflated by roughly $5/trade × 1800 trades = ~$9,000 over the year.
**Root cause:** Copy-paste from a single-contract prototype. Nobody questioned why commission was so low.
**Lesson:** Always verify financial calculations against a manual example. $1.29 × 2 sides × 5 contracts = $12.90. Period.

### 1.2 Intra-Bar Stops Weren't Using Wicks (CRITICAL)

**Commit:** `a6df582b`
**What was wrong:** Stop-loss checks only used `bar.close`. A bar could wick through your stop (bar_low < stop_price for longs) and recover, but the bot never noticed.
**Impact:** Backtest was ignoring ~15-20% of stop-outs. Trades that should have been stopped out continued running, making the system look more profitable than reality.
**Root cause:** The executor's `update()` method didn't accept `bar_high`/`bar_low` parameters. Nobody passed wick data.
**Lesson:** Always check: "Is my stop execution using the WORST price the market hit, or just the close?" For futures backtesting, intra-bar wicks are non-negotiable.

### 1.3 C3 Double-Counting PnL (CRITICAL)

**Commit:** `a6df582b`
**What was wrong:** When C3 was blocked (C1 lost), C3's PnL was removed from the trade total, but the exit slippage for C3 was still being charged. The "savings" calculation was also adding C3's commission back instead of subtracting it.
**Impact:** C3-blocked trades had inconsistent accounting. Some showed savings that were inflated by the commission sign error.
**Root cause:** Complex multi-step adjustment logic where removing a leg's contribution requires touching 3 different numbers (gross PnL, commission, slippage) and getting the signs right on all of them.
**Lesson:** When adjusting PnL for blocked legs, write it as a single formula and verify the signs: `adjusted = total - c3_gross - c3_commission + c3_slippage_refund`.

### 1.4 ATR Safety Gate Blocking Active Position Management (CRITICAL)

**Commit:** `dfb01d99`
**What was wrong:** The ATR safety gate (`if atr_14 <= 0: return None`) was at line 340 in main.py. The active position management (`executor.update()`) was at line 388. If ATR dropped to 0 mid-trade, the function returned before ever managing the active position — no stops, no trailing, no exits.
**Impact:** Any trade active when ATR went to 0 became a ghost position. It couldn't exit, couldn't trail, couldn't do anything. It just sat there accumulating risk.
**Root cause:** The original code was written linearly (check indicators → manage position → check entries) without considering that position management must ALWAYS run regardless of indicator state.
**Fix applied:** Moved position management to step 1b (BEFORE the safety gate). Gate now only blocks NEW entries.
**Lesson:** Position management is unconditional. Never gate it behind any indicator check. The flow should always be: (1) manage existing positions, (2) check if conditions allow new entries, (3) enter if allowed.

### 1.5 C3 Fallback Parameters Pointed to C2 Config (HIGH)

**Commit:** `dfb01d99`
**What was wrong:** Six `getattr()` fallbacks for C3 parameters were referencing C2 config fields instead of literal defaults:
- `c3_trailing_atr_multiplier` fell back to `cfg.c2_trailing_atr_multiplier` (2.0) instead of 3.0
- `c3_max_target_points` fell back to `cfg.c2_max_target_points` (150.0) instead of 300.0
- `c3_time_stop_minutes` fell back to `cfg.c2_time_stop_minutes` (120) instead of 240
- `c2_be_variant` fell back to `"D"` instead of `"B"`

**Impact:** If the config file was missing any C3 field, C3 silently degraded to C2's tighter parameters. The trail runner became a structure trade. 3.0× ATR trail became 2.0×. 300pt cap became 150pt. 4-hour timeout became 2 hours.
**Root cause:** When adding C3 parameters, someone used "copy C2's getattr, change the field name" but forgot to change the fallback value.
**Lesson:** Never use another config field as a fallback. Use a literal. `getattr(cfg, 'c3_field', 3.0)` not `getattr(cfg, 'c3_field', cfg.c2_field)`.

### 1.6 C3 Contracts Hardcoded to 3 in Orchestrator (MEDIUM)

**Commit:** `dfb01d99`
**What was wrong:** Two locations in orchestrator.py had `contracts=3` hardcoded instead of reading `cfg.c3_contracts`.
**Impact:** If someone changed c3_contracts in config (e.g., to 2 for lower risk), the orchestrator would still use 3. Backtest and live would diverge.
**Lesson:** Search the entire codebase for magic numbers after adding any config field. `grep -rn "contracts.*3" execution/`.

### 1.7 C3_DELAYED_ENTRY Was Module-Level Constant in Backtest (MEDIUM)

**Commit:** `dfb01d99`
**What was wrong:** `full_backtest.py` used a module-level `C3_DELAYED_ENTRY = True` constant instead of reading from the config's `scale_out.c3_delayed_entry_enabled` field.
**Impact:** Couldn't toggle C3 delayed entry from config for backtest runs. The feature was always on regardless of config.
**Lesson:** No module-level behavioral constants. Everything configurable must be read from config at runtime.

## PART 3: ARCHITECTURAL WEAKNESSES

These aren't bugs per se, but structural issues that make the codebase fragile and prone to future bugs.

### 3.1 Backtest/Live Code Duplication (THE #1 WEAKNESS)

**The problem:** `scale_out_executor.py` (backtest) and `orchestrator.py` (live) implement the SAME trading logic independently. Every parameter, every threshold, every exit condition exists in TWO places. When you fix one, you must fix the other. When you forget, you get silent divergence (see bugs 2.1 and 2.2 above).

**Files affected:**
- C1 time exit logic: executor line 617-647 vs orchestrator line 842-920
- C2 time stop: executor line 847 vs orchestrator line 1106
- C3 trailing logic: executor line 886-900 vs orchestrator line 1121-1145
- Breakeven logic: executor line 680-716 vs orchestrator line 941-985
- C3 delayed entry: executor line 695-705 vs orchestrator line 986-1002

**Why it matters:** Every future change must be made in two places. Every optimization must be verified in both. This is the single biggest source of bugs in the codebase.

**Recommended fix:** Extract a shared `TradeLogic` class that both executor and orchestrator call. Single source of truth for all thresholds, exit conditions, and state transitions.

### 3.2 Config Validation Is Nonexistent

**The problem:** There are 13+ scale-out config fields that are read via `getattr()` with fallback defaults. None are validated on startup. If the config file is missing a field, has a typo, or has an unreasonable value, the system silently uses the default.

**Config fields with no validation:**
1. `c1_time_exit_bars` (default varies)
2. `c1_max_bars_fallback` (default varies)
3. `c1_profit_threshold_pts` (default 3.0)
4. `c2_time_stop_bars` (default 35)
5. `c2_be_variant` (default "B")
6. `c2_be_delay_multiplier` (default 1.5)
7. `c2_breakeven_buffer_points` (default varies)
8. `c3_trailing_atr_multiplier` (default 3.0)
9. `c3_max_target_points` (default 300.0)
10. `c3_time_stop_minutes` (default 240)
11. `c3_contracts` (default 3)
12. `c3_delayed_entry_enabled` (default True)
13. `c2_structural_target_source` (default varies)

**Recommended fix:** Add a `validate_scale_out_config()` function that runs on startup, checks all fields exist, and logs warnings for any missing with their default values.

### 3.3 No Unified Slippage Model

**The problem:** `full_backtest.py` uses fixed slippage (0.50pt RTH, 1.00pt ETH). `replay_simulator.py` uses a dynamic model with volume spikes, news windows, and stochastic variance. The same trade produces different slippage in each backtester.

**Impact:** Results from the two backtesters aren't comparable. If you optimize on one and validate on the other, the slippage difference can flip marginal trades.

**Recommended fix:** Move slippage logic to a shared `SlippageModel` class. Both backtesters import the same model.

### 3.4 Race Conditions in Live Orchestrator

**The problem:** The orchestrator uses a single `asyncio.Lock()` for bar processing, but:
- The reconciliation loop accesses `_c3_tracking` and `_leg_state` without acquiring the lock
- Dictionary operations on `_c3_tracking` are not atomic
- If reconciliation detects a fill while bar processing is checking targets, state can become inconsistent

**Impact:** In live trading, rare but possible: double-exit a position, or miss a fill notification.

**Recommended fix:** All state mutations to `_c3_tracking`, `_leg_state`, and position data must be inside the `_bar_lock`. Reconciliation must acquire the lock before modifying state.

### 3.5 Sweep Event Not Cleared After Trade Close

**File:** `full_backtest.py` lines 954-956
**What's wrong:** `_sweep_event` and `_sweep_fvg` are only cleared on session boundary, not when a trade closes. If a trade closes mid-session, the next signal could use stale sweep data from the previous trade.

## PART 4: STRENGTHS

Not everything is broken. Here's what's working well:

### 4.1 Causal Replay Engine Design
The bar processing order (HTF feed → pending entry execution → position management → feature computation → signal generation) is fundamentally sound. Signals on bar N execute on bar N+1 open. No future data leaks in the main signal path.

### 4.2 C3 Delayed Entry Concept
The idea of only keeping C3 open when C1 confirms profitability is excellent risk management. When it works correctly, it saves ~$50K in avoided losses over the test year. The implementation just needs the accounting fixes listed above.

### 4.3 Intra-Bar Wick Execution (After Fix)
Now that bar_high/bar_low are passed to the executor, stops properly check against the worst intra-bar price. This is critical for realistic futures backtesting and was a major improvement.

### 4.4 HTF Bias Gating
The two-timeframe consensus approach (5m + 15m) with hysteresis and flip confirmation prevents whipsaw from rapid bias changes. The design is well-thought-out.

### 4.5 Maintenance Window Protection
The 4:30 PM ET entry cutoff and 4:50 PM forced flatten is a solid safety feature. It prevents new entries during the dangerous pre-maintenance period and guarantees no positions carry through the daily halt.

### 4.6 Commission Model (After Fix)
The per-contract, per-side commission model is now correct: $1.29/contract/side × 2 sides × N contracts. Applied at entry for the full round-trip. Consistent across all paths.

## PART 7: ERROR PATTERNS — WHAT KEEPS GOING WRONG

These are the recurring mistake patterns we need to break:

### Pattern 1: "Copy C2, Change the Name"
**Occurrences:** C3 fallbacks (×6), C3 contracts (×2), C2 time stop bars
**The mistake:** When adding a new parameter for C3, copying the C2 getattr line and only changing the field name, not the fallback value.
**Prevention:** After adding ANY config field, grep the entire codebase for both the old and new field names. Verify every fallback is a literal, not a reference to another field.

### Pattern 2: "Works in Backtest, Different in Live"
**Occurrences:** C2 time stop (35 vs 20), C1 profit threshold (config vs hardcode)
**The mistake:** Making a fix in the backtest executor but not the live orchestrator, or vice versa.
**Prevention:** Every change to trade logic MUST touch BOTH scale_out_executor.py AND orchestrator.py. Add a checklist to the PR template.

### Pattern 3: "The Safety Gate Blocks Everything"
**Occurrences:** ATR gate blocking position management
**The mistake:** Placing a safety check (return early) before essential logic that must always run.
**Prevention:** Position management is NEVER conditional. Design the flow as: (1) unconditional management, (2) conditional gating for new entries only.

### Pattern 4: "The Sign is Wrong"
**Occurrences:** C3 commission added instead of subtracted
**The mistake:** Multi-step PnL adjustments where one term has the wrong sign.
**Prevention:** Write the full formula on one line with comments: `adjusted = total - c3_gross - c3_commission + c3_slippage_refund # (minus costs, plus refunds)`

### Pattern 5: "Module-Level Constants Instead of Config"
**Occurrences:** C3_DELAYED_ENTRY, FVG tracking params, maintenance window times
**The mistake:** Hardcoding behavioral parameters as module-level constants that can't be changed without editing code.
**Prevention:** If a value affects trading behavior, it MUST be in the config file. Module-level constants are only for truly fixed values (POINT_VALUE = 2.0, etc.).

## PART 9: V1.3.5b SESSION — NEW BUGS & LESSONS (March 14, 2026)

### 1.8 Verification Check Too Strict for Variable Commission (LOW)

**What happened:** Commission verification check expects $12.90 per trade (5 contracts), but C3-blocked trades correctly pay $5.16 (2 contracts). Check reports [FAIL] on valid behavior.
**Impact:** Cosmetic — causes false alarm on every backtest run. No financial impact.
**Lesson:** Verification checks must account for variable contract counts. The check should validate: C3-blocked → $5.16, C3-entered → $12.90.

### 1.9 replay_simulator.py Missed During Time Window Change (MEDIUM)

**What happened:** Three agents made changes in parallel. The time window agent updated 6 files (main.py, orchestrator.py, full_backtest.py, tradovate_paper.py, data_adapter.py, constants.py) but MISSED replay_simulator.py. The cross-verification agent caught it.
**Impact:** Would have caused replay_simulator to use 4:30 PM cutoff while everything else uses 3:30 PM.
**Lesson:** When changing a behavioral constant, ALWAYS grep the entire codebase for the old value. `grep -rn "16.*30\|4:30\|SESSION_CLOSE" --include="*.py"` would have caught it.

### 1.10 dashboard/data_adapter.py Had Second Hardcoded Time Check (LOW)

**What happened:** data_adapter.py line 281 had a hardcoded `(prev_dt.hour == 16 and prev_dt.minute >= 30)` for maintenance gap detection, separate from the constants import. The time window agent updated line 41 but missed line 281.
**Impact:** Dashboard gap classification would be wrong. No trading impact.
**Lesson:** After changing a time constant, grep for the NUMERIC values too, not just the constant name: `grep -rn "== 16" --include="*.py"`.

## PART 10: STRATEGY LESSONS FROM BACKTEST ANALYSIS

### Lesson 1: Tight Stops Are Noise Traps
**Discovery:** Stops under 12pt lost $1,003 over 511 trades in V1.3.5. After adding the 12pt floor, the min-stop gate became the 2nd most valuable filter in the system (blocked 3,469 signals, saved $22,327).
**Rule:** Never enter with a stop under 12 points on MNQ. The 20-25pt stop range is the sweet spot ($25-31/trade avg).

### Lesson 2: Ranging Regime Is Below Breakeven
**Discovery:** Ranging has 32.1% WR but needs 32.5% to break even. C1+C2 lose $10,698 in ranging; C3 barely saves it with $10,201. Ranging LONGS are especially toxic (28.6% WR, -$2,032).
**Rule:** Either skip ranging entirely or block ranging longs. The system is a trend-follower — forcing it into ranges destroys edge.

### Lesson 3: 35% Win Rate Is Fine — It's the R:R That Matters
**Discovery:** Breakeven WR for this system is 30.7% (given 2.255:1 R:R). The actual 35.1% gives 4.4pp of edge. This matches professional trend-following systems.
**Rule:** Never optimize for higher win rate at the expense of R:R. Raising WR from 35% to 45% means nothing if avg winner drops from $222 to $150.

### Lesson 4: 80% of Monthly Profit Comes From ~22 Big Winners
**Discovery:** Each profitable month is driven by 20-30 trades making $200+. Small winners barely cover small losers. Miss those big winners and the month is negative.
**Rule:** Never add filters that reduce "time in market" during trending conditions. The big winners are C3 runners in trending regimes — protect access to those at all costs.

### Lesson 5: Regime Detection Has 75% Flip Rate
**Discovery:** The regime detector changes regime on 75% of trades. 98.8% of flips reverse within 5 trades. The detector is oscillating, not detecting real transitions.
**Rule:** Don't trust single-bar regime changes. Any regime-based filter should require 3+ bars of confirmation before acting. Consider reducing ATR lookback from 20 to 12 bars.

### Lesson 6: Late-Afternoon Entries Lose Money
**Discovery:** 3-4 PM entries cost $2,047 in V1.3.5. Moved entry cutoff from 4:30 PM to 3:30 PM to give positions 80 minutes to close before maintenance.
**Rule:** No new entries after 3:30 PM ET during RTH. Active positions must close by 4:50 PM maintenance flatten.

### Lesson 7: Parallel Agent Changes Need Cross-Verification
**Discovery:** When 4 agents work in parallel, they inevitably miss files or create inconsistencies. The cross-verification agent found 2 files the other agents missed.
**Rule:** After any multi-agent parallel edit session, ALWAYS run a separate verification agent that reads every modified file and checks for consistency, conflicts, and missing changes.
