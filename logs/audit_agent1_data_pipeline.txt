1. CONTRACT RESOLUTION CHAIN
------------------------------------------------------------------------
File: Broker/ibkr_client_portal.py

The IBKRClient uses a 4-strategy fallback for contract resolution:
  Strategy 1: /iserver/secdef/search — searches by symbol, filters for FUT
  Strategy 2: /trsrv/futures — queries tradeable futures by symbol
  Strategy 3: /iserver/secdef/info — uses conid from strategy 1/2 results
  Strategy 4: Cached contract from previous session

Each strategy:
  - Constructs valid HTTP POST/GET requests via _post()/_get() helpers
  - Parses JSON responses with proper type checking (isinstance checks)
  - Has try/except wrapping that logs errors and falls through to next strategy
  - Never crashes — worst case returns None, which is caught by the caller

Finding 1A [LOW]: Strategy 1 (line ~1218) has a non-FUT fallback that could
match non-futures securities if the FUT section is missing. Unlikely in
practice since MNQ always has FUT results.

Finding 1B [LOW]: Strategy 3 (line ~1331) uses string date comparison for
expiry filtering. Works correctly for ISO date strings but fragile if
format ever changes.

Verdict: Contract resolution is ROBUST with proper error handling.

2. CANDLE AGGREGATOR OUTPUT vs process_bar() INPUT
------------------------------------------------------------------------
Aggregator File: signals/aggregator.py
Consumer File: main.py (process_bar at ~line 350)

CandleAggregator outputs a bar dict/object with fields:
  - timestamp (datetime) — candle close time
  - open (float) — open price
  - high (float) — highest price in candle
  - low (float) — lowest price in candle
  - close (float) — close price
  - volume (int/float) — total volume

process_bar() expects:
  - bar.timestamp (datetime)
  - bar.open, bar.high, bar.low, bar.close (float)
  - bar.volume (int/float)

Field-by-field comparison: ALL FIELDS MATCH in name, type, and semantics.

Finding 2A [MEDIUM]: In Client Portal HTTP polling mode, "volume" is actually
a snapshot count (number of snapshots aggregated), NOT real trade volume.
This is because the Client Portal REST API returns market data snapshots,
not tick-by-tick data. The CandleAggregator counts snapshots, not actual
volume. This means volume-based indicators (volume spike detection in
sweep detector, volume profile) will be unreliable in HTTP polling mode.

Finding 2B [HIGH]: The TWS path (Broker/ibkr_client.py via ib_insync)
delivers 5-second bars by default (line 239: reqRealTimeBars), NOT 2-minute
candles. There is no CandleAggregator in the TWS data path to aggregate
these into 2-minute candles. If the TWS path is used instead of Client
Portal, the data format will be wrong.

Verdict: Client Portal path is CORRECT. TWS path has format mismatch.

3. SESSION DETECTION
------------------------------------------------------------------------
File: Broker/ibkr_client_portal.py — get_session_type()

RTH: 09:30-16:00 ET — CORRECT
ETH: 18:00-09:29 ET — CORRECT
Dead zone: 16:00-18:00 — classified as ETH (technically "closed" period)

Timezone handling: Uses ZoneInfo("America/New_York") consistently.
DST-safe per security audit fix. All timestamps converted to ET before
session classification.

Finding 3A [INFO]: 16:00-18:00 classified as ETH rather than "closed".
This is acceptable — no trades should fire during this period anyway since
the exchange is closed (no data flowing). No functional impact.

Verdict: Session detection is CORRECT and DST-safe.

4. DATA QUALITY CHECKS
------------------------------------------------------------------------
File: data_pipeline/pipeline.py

Zero-volume rejection: The pipeline does NOT explicitly reject zero-volume
candles from the IBKR real-time feed. However, zero-volume bars during
live trading (outside closed periods) are extremely rare for MNQ.

High < Low rejection: The pipeline does NOT explicitly check for high < low
candles. The 4-layer NaN/Inf defense (from security audit) catches NaN/Inf
but not logically invalid price relationships.

Gap detection: Gap detection (>2 min between candles) is implemented in the
CandleAggregator with timestamp tracking and logging.

Finding 4A [LOW]: TradingView CSV import (pipeline.py line ~314) accepts
zero-volume bars without filtering. Not relevant for live trading, only
backtesting.

Verdict: Adequate for live trading; edge cases in CSV import.

5. HISTORICAL BACKFILL
------------------------------------------------------------------------
File: scripts/run_ibkr.py — IBKRRunner

ATR-14 warmup requires minimum 14 bars (2-minute candles = 28 minutes).
Volume average (20-bar) requires 20 bars = 40 minutes.
HTF engine needs at least 1 bar per timeframe (5m, 15m, 30m, 1H, 4H, 1D).

The IBKRRunner implements a warmup phase (run_ibkr.py line ~370):
  - _warmup_bar_count tracks bars received
  - Trading is suppressed until warmup completes
  - WARMUP_BARS constant controls the minimum

Finding 5A [MEDIUM]: During warmup, bars are fed to the feature engine
but trading is suppressed. However, the backfill comes from real-time
bars (not historical request), meaning the system must wait 28+ minutes
after startup before ATR-14 is primed. A historical backfill request
(e.g., /iserver/marketdata/history) could prime indicators instantly.

Mitigated by: The HTF gate blocks all trades when HTF data is stale
(main.py lines 609-617), providing an additional safety net during warmup.

Verdict: ACCEPTABLE — warmup phase exists but relies on real-time bars.

6. WEBSOCKET vs HTTP FALLBACK
------------------------------------------------------------------------
File: Broker/ibkr_client_portal.py

Primary: HTTP snapshot polling (not WebSocket) via _poll_loop()
  - poll_interval_seconds configurable (default varies by config)
  - _connected flag controls the loop
  - Errors logged and loop continues (line ~1500)

WebSocket: NOT the primary method. The Client Portal Gateway uses
REST + HTTP polling for market data snapshots.

TWS path (Broker/ibkr_client.py): Uses ib_insync WebSocket connection
with automatic reconnection built into the ib_insync library.

Finding 6A [INFO]: No explicit WebSocket-to-HTTP failover exists because
the Client Portal path uses HTTP polling as its primary method. The
ib_insync TWS path has its own reconnection logic.

Finding 6B [INFO]: The async callback in _poll_loop (line 1497) is
correctly awaited.

Finding 6C [INFO]: SSL verification disabled (CERT_NONE) for localhost
gateway connections. This is correct — the IBKR Client Portal Gateway
runs on localhost with a self-signed certificate.

Verdict: HTTP polling is the primary method; no failover needed.

ADDITIONAL FINDINGS
========================================================================

Finding A1 [HIGH]: No CandleAggregator exists in the TWS data path.
If the system uses the TWS adapter (ibkr_client.py) instead of the Client
Portal adapter, raw 5-second bars would be passed directly to process_bar()
without aggregation into 2-minute candles.

Finding A2 [LOW]: MIN_RR_RATIO used in main.py line 699 is not defined
in config/constants.py. It appears to be defined locally in main.py.
Should be in constants.py for single-source-of-truth consistency.

Finding A3 [MEDIUM]: Volume data quality in Client Portal mode — the
"volume" field in snapshots is not true trade volume but a snapshot count.
Volume-dependent signals (sweep volume spike detection) will produce
different results in live trading vs backtesting.
